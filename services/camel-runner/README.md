# Camel Runner Service

The `camel-runner` service is a **Dockerized Camel JBang execution environment** isolated from the main TypeScript application.

## Purpose

- Accept YAML route artifacts generated by the Node.js API
- Execute them via `camel run` (Camel JBang)
- Report execution status back to the API

## Why JBang and not Camel K?

Camel K requires a running Kubernetes cluster and an Operator. At this stage the hardest product problems are template design, mapping intelligence, and Dev/Test/Prod promotion — not runtime Kubernetes isolation. JBang lets us execute generated routes without any Java build toolchain on the host machine.

Camel K will be considered later when runtime isolation and multi-tenant scaling become real operational needs. See the main README for the full rationale.

## How routes flow

```
Control Plane (TypeScript)
  │
  ├─ generates YAML artifact  (packages/camel)
  ├─ writes to shared volume   (/app/routes/<integration-id>.yaml)
  └─ POST /run  →  camel-runner executes it via `camel run`
```

## Local execution (without Docker)

If you have Camel JBang installed:

```bash
# Install once
curl -Ls https://sh.jbang.dev | bash
jbang app install camel@apache/camel

# Run a route
camel run routes/sample-rest-to-rest.yaml
```

## Container restart & concurrency

The V1 runner handles one execution at a time. Concurrency and restart handling will be addressed when moving toward a production cloud deployment.

## Environment variables

| Variable | Default | Purpose |
|---|---|---|
| `RUNNER_PORT` | `8080` | Port the HTTP dispatcher listens on |

## HTTP API

| Method | Path | Body | Description |
|---|---|---|---|
| `GET` | `/health` | — | Liveness check |
| `POST` | `/run` | `{"route": "/app/routes/my-route.yaml"}` | Execute a route file |
| `POST` | `/preview-run` | `{"route": "/app/routes/my-preview-route.yaml"}` | Execute a one-shot preview route and return extracted mapped payload |
